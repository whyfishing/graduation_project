## 2. 设计目标

1. 支撑网易云单平台数据采集、查询、分析和可视化闭环。  
2. 用“实体表 + 映射表 + 分析结果表”替代冗余大宽表。  
3. 保证核心实体主键唯一、关系可追溯、扩展成本低。  
4. 为后续多平台扩展预留字段，不影响当前单平台落地。

## 3. 设计范围与约束

### 3.1 范围

1. 排行榜推荐与榜单歌曲明细。  
2. 歌单推荐与歌单歌曲明细。  
3. 歌手推荐与歌手代表歌曲。  
4. 歌曲标签、听众画像、歌单画像、歌手画像、榜单画像。  
5. 采集任务与数据处理任务日志。

### 3.2 约束

1. 数据源以网易云网页结构和公开可见数据为准。  
2. 仅采集合法公开信息，不存储敏感隐私。  
3. 用户相关数据用于样本分析，需具备最小化采集原则。  

## 4. 逻辑模型（ER 文字版）

1. `songs` 与 `artists` 为多对多，通过 `artist_songs` 关联。  
2. `songs` 与 `playlists` 为多对多，通过 `playlist_songs` 关联。  
3. `songs` 与 `ranks` 为多对多，通过 `rank_songs` 关联。  
4. `users` 与 `songs` 为多对多，通过 `user_recent_songs` 关联。  
5. `songs` 与 `song_tags` 为一对多。  
6. `users`、`playlists`、`artists`、`ranks` 分别与画像表一对一或一对多（按版本）。

## 5. 命名与通用规范

1. 库名：`music_system`。  
2. 表名：小写下划线，按领域命名。  
3. 主键：业务主键优先（如 `song_id`），必要时补自增 `id`。  
4. 时间字段：统一 `datetime`，命名 `created_at/updated_at/crawl_time`。  
5. 字符集：`utf8mb4`，排序规则 `utf8mb4_unicode_ci`。

## 7. 主外键与完整性策略

1. 开发阶段可先使用逻辑外键（应用层保证），减少采集写入阻塞。  
2. 稳定后为核心关系补外键：`playlist_songs.song_id -> songs.song_id` 等。  
3. 映射表强制唯一键，防止重复关系。  
4. 画像和标签表按版本保留历史，不覆盖旧版本。

## 8. 分区与归档建议

1. `rank_songs`、`user_recent_songs`、`sys_task_log` 可按月归档。  
2. 高频查询表建议按 `crawl_time` 增加组合索引。  
3. 历史画像按版本和时间保留最近 N 版，旧版归档。

## 9. 核心查询场景与SQL样例

### 9.1 查询某榜单最新 TopN

```sql
SELECT rs.rank_position, s.song_id, s.song_name, a.artist_name
FROM rank_songs rs
JOIN songs s ON rs.song_id = s.song_id
LEFT JOIN artist_songs ars ON s.song_id = ars.song_id
LEFT JOIN artists a ON ars.artist_id = a.artist_id
WHERE rs.rank_id = ?
  AND rs.crawl_time = (
      SELECT MAX(crawl_time) FROM rank_songs WHERE rank_id = ?
  )
ORDER BY rs.rank_position
LIMIT 50;
```

### 9.2 查询歌单画像所需标签分布

```sql
SELECT st.tag_name, COUNT(*) AS tag_cnt
FROM playlist_songs ps
JOIN song_tags st ON ps.song_id = st.song_id
WHERE ps.playlist_id = ?
GROUP BY st.tag_name
ORDER BY tag_cnt DESC;
```

### 9.3 查询用户最近听歌与画像

```sql
SELECT urs.play_order, s.song_name, up.profile_text
FROM user_recent_songs urs
JOIN songs s ON urs.song_id = s.song_id
LEFT JOIN user_profiles up ON up.user_id = urs.user_id
WHERE urs.user_id = ?
ORDER BY urs.play_order ASC;
```

## 10. 数据质量与审计要求

1. 主实体唯一ID不能为空，非法ID记录进异常日志。  
2. 关键字段缺失率需统计：song_name、artist_name、playlist_name。  
3. 采集批次必须记录到 `sys_task_log`，失败可追踪。  
4. 标签与画像需记录 `source_model/version`，支持复算对比。

## 11. 安全与合规

1. 仅存储公开可见信息，不存储敏感账号信息。  
2. 用户画像仅用于研究分析，展示时脱敏。  
3. 外部AI接口调用需记录调用摘要，不记录密钥。  

## 12. 实施顺序建议

1. 第一阶段：完成实体表 + 映射表 + 采集落库。  
2. 第二阶段：完成 `song_tags` 与画像结果表。  
3. 第三阶段：补齐约束、索引优化、归档策略。
