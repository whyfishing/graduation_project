## 2. 执行摘要（Executive Summary）

本系统采用“离线批处理 + 准实时流处理 + 检索服务 + 可视化展示”的组合架构，重点满足毕业设计场景中的可实现性、可演示性和可验证性。当前已确认技术栈为 Vue 3 + Spring Boot + MySQL + Kafka + FAISS + Docker Compose。核心决策如下：

1. 数据处理以 Spark 为核心，离线与流式统一技术栈，降低学习与实现成本。  
2. 架构选择“模块化单体 + 数据处理子系统”而非微服务，减少运维复杂度。  
3. 数据分层采用 ODS/DWD/DWS/ADS，保证分析结果可追溯。  
4. 听歌识曲与哼唱搜歌采用“特征提取 + 向量检索”路线，优先实现可演示基线效果。  
5. 部署采用单机 Docker Compose，确保环境可复制与演示稳定性。

## 3. 架构驱动与约束

### 3.1 核心功能驱动

1. 多平台音乐数据采集与清洗治理。  
2. 实时榜单（分钟级更新）与多维统计。  
3. 音乐分类、听众分群、群体画像。  
4. 听歌识曲、哼唱搜歌、相似歌曲检索。  
5. 可视化看板与管理后台。

### 3.2 非功能驱动（毕设目标）

1. 可演示：核心链路端到端跑通。  
2. 可维护：模块边界清晰，支持独立调试。  
3. 可验证：指标、日志、数据血缘可追踪。  
4. 成本可控：尽量复用开源组件，部署简化。

### 3.3 关键约束

1. 不追求企业级高可用与超大规模吞吐。  
2. 数据量以展示效果为主，兼顾真实性。  
3. 仅使用公开数据并遵守合规要求。  
4. 开发周期受毕业设计时间约束。

## 4. 总体架构设计

### 4.1 架构风格

采用“分层架构 + 事件驱动补充”的混合方案：

1. 接入层：采集任务、音频上传、查询请求。  
2. 处理层：Spark 离线/流式计算、特征提取、分群训练。  
3. 服务层：排行榜服务、画像服务、检索服务、管理服务。  
4. 展示层：分析看板与功能页面。  
5. 存储层：关系库 + 文件存储 + 缓存（可选）+ 消息队列（可选）。

### 4.2 C4-Context（文字化）

系统对外部参与方关系：

1. 外部数据平台（QQ音乐/酷狗/抖音）向系统提供公开元数据来源。  
2. 管理员通过后台配置采集任务与参数。  
3. 分析用户通过 Web 页面查看榜单、画像和趋势。  
4. 演示用户上传音频片段体验识曲/哼唱搜歌。  
5. 系统输出图表、榜单、检索结果和任务日志。

### 4.3 C4-Container（容器级）

1. Web 前端容器：展示看板与交互页面。  
2. API 后端容器：对外接口聚合、权限、业务编排。  
3. 数据采集容器：定时采集、清洗入库。  
4. Spark 计算容器：批处理 + Structured Streaming。  
5. 检索容器：音频特征提取与向量查询。  
6. 数据库容器：结构化数据、结果快照、任务元数据。  
7. 文件存储容器：原始数据、音频样本、中间结果。  
8. 可视化服务容器：图表查询与数据聚合接口。

## 5. 模块设计与职责

### 5.1 数据采集与治理模块

职责：

1. 执行多平台爬取任务。  
2. 进行去重、标准化、异常标记。  
3. 维护跨平台实体映射（song_id 统一）。  
4. 输出 ODS 与 DWD 层数据。

输入：平台公开页面/API数据。  
输出：标准化歌曲、歌手、行为记录。

### 5.2 实时计算模块（Spark Streaming）

职责：

1. 消费新增行为数据流。  
2. 按窗口计算榜单指标（播放量、增速、互动度）。  
3. 生成 TopN 快照并写入服务层存储。  
4. 输出实时告警指标（可选）。

输入：行为增量流。  
输出：日/周/月榜、趋势指标。

### 5.3 离线分析与建模模块（Spark SQL / MLlib）

职责：

1. 执行分类、分群、画像计算。  
2. 训练和更新听众分群模型。  
3. 输出可视化数据集（ADS层）。  
4. 支持周期性重算与版本追踪。

输入：DWD 数据、特征表。  
输出：画像标签、群体对比、分类结果。

### 5.4 音频检索模块

职责：

1. 音频预处理（采样率统一、降噪、裁剪）。  
2. 特征提取（MFCC、Chroma、Tempo、Embedding）。  
3. 向量索引构建与检索。  
4. 识曲/哼唱返回 TopK 及置信度。

输入：音频片段或哼唱。  
输出：候选歌曲列表、相似度评分。

### 5.5 API 网关与业务服务模块

职责：

1. 统一提供 REST API。  
2. 聚合排行榜、画像、检索结果。  
3. 鉴权与角色权限控制。  
4. 记录审计日志与任务状态。

输入：前端请求。  
输出：标准 JSON 响应。

### 5.6 可视化与后台模块

职责：

1. 榜单、趋势、分类、画像、检索结果展示。  
2. 多维筛选与联动分析。  
3. 任务监控与参数配置。  
4. 数据导出与答辩展示模式。

## 6. 数据架构设计

### 6.1 分层模型

1. ODS（原始层）：原始采集记录，保留来源与抓取时间。  
2. DWD（明细层）：清洗标准化后的明细事实表。  
3. DWS（汇总层）：按主题汇总（歌曲、歌手、用户分群）。  
4. ADS（应用层）：直接供前端和接口使用的数据集。

### 6.2 核心数据表建议

1. dim_song：歌曲维表。  
2. dim_artist：歌手维表。  
3. fact_play_event：播放行为事实表。  
4. fact_rank_snapshot：榜单快照表。  
5. fact_user_segment：用户分群结果表。  
6. fact_song_embedding：歌曲向量特征表。  
7. sys_task_log：任务执行日志表。

### 6.3 数据流（关键链路）

1. 采集任务将平台数据写入 ODS。  
2. 清洗任务从 ODS 生成 DWD。  
3. Spark 批处理从 DWD 生成 DWS/ADS。  
4. Streaming 从增量行为实时更新 rank_snapshot。  
5. 检索模块读取 embedding 并返回相似结果。  
6. API 聚合 ADS 与实时结果返回前端。

## 7. 接口设计原则

### 7.1 API 风格

1. 统一 REST 风格。  
2. URL 版本化（/api/v1/...）。  
3. 响应结构统一（code、message、data、trace_id）。

### 7.2 错误处理标准

1. 参数错误：400。  
2. 鉴权失败：401/403。  
3. 资源不存在：404。  
4. 服务异常：500。  
5. 检索超时：504（可选）。

### 7.3 稳定性策略

1. 采集重试与限速。  
2. 计算任务失败补偿。  
3. 检索接口超时控制与降级（返回缓存候选）。

## 8. 技术选型对比与决策建议

本节给出“毕设可落地”优先的选型，不追求企业级复杂度。
本节用于保留备选方案对比，最终定稿方案以第 9 章和第 13 章为准。

### 8.1 前端技术选型

| 方案 | 优点 | 缺点 | 结论建议 |
|---|---|---|---|
| Vue 3 + Vite + ECharts | 上手快、生态成熟、图表集成简单 | TypeScript 约束可选，规范需自控 | 推荐（毕设首选） |
| React + Vite + Recharts/ECharts | 生态大、组件丰富 | 状态管理和规范成本稍高 | 若你更熟 React 可选 |
| Streamlit | 极快搭建演示页 | 页面定制和交互复杂度受限 | 快速原型可选，不建议主方案 |

默认建议：Vue 3 + ECharts。

### 8.2 后端 API 技术选型

| 方案 | 优点 | 缺点 | 结论建议 |
|---|---|---|---|
| FastAPI（Python） | 与数据处理同语言、开发快、文档自动生成 | 高并发不是强项（毕设可接受） | 推荐（毕设首选） |
| Flask | 轻量简单、学习成本低 | 工程化能力弱于 FastAPI | 可用但次优 |
| Spring Boot（Java） | 工程化强、稳定性好 | 学习和开发成本高 | 除非你熟 Java，不建议毕设首选 |

默认建议：FastAPI。

### 8.3 数据库选型

| 方案 | 优点 | 缺点 | 结论建议 |
|---|---|---|---|
| MySQL | 熟悉度高、部署简单、足够支撑毕设 | 分析查询性能一般 | 推荐（元数据与结果库） |
| PostgreSQL | SQL能力强、扩展好 | 你若不熟学习成本略高 | 可作为 MySQL 替代 |
| SQLite | 轻量免运维 | 并发和扩展有限 | 仅适合极简 Demo |

默认建议：MySQL。

### 8.4 消息队列选型（可选）

| 方案 | 优点 | 缺点 | 结论建议 |
|---|---|---|---|
| Kafka | 与 Spark Streaming 契合、生态成熟 | 部署与运维相对复杂 | 有时间精力可选 |
| RabbitMQ | 部署相对容易、学习曲线平缓 | 大吞吐流处理生态不如 Kafka | 毕设友好备选 |
| 无消息队列（DB/文件增量模拟流） | 最简单，开发快 | 真实性与扩展性较弱 | 时间紧可选 |

默认建议：时间充足用 Kafka；时间紧使用“无MQ增量模拟流”。

### 8.5 向量检索选型

| 方案 | 优点 | 缺点 | 结论建议 |
|---|---|---|---|
| FAISS | 检索性能好、学术与工程常用 | 依赖安装可能稍复杂 | 推荐（首选） |
| Annoy | 轻量、上手快 | 准确率与功能不如 FAISS 全面 | 备选 |
| 纯余弦暴力检索 | 实现最简单 | 数据稍大时慢 | 小样本可用保底方案 |

默认建议：FAISS，保留暴力检索作为 fallback。

### 8.6 调度与工作流选型

| 方案 | 优点 | 缺点 | 结论建议 |
|---|---|---|---|
| Airflow | 调度可视化强、DAG清晰 | 部署和配置较重 | 功能完整但偏重 |
| APScheduler + Cron | 极简、够用 | 可视化和编排能力弱 | 推荐（毕设首选） |
| 手动触发脚本 | 开发最快 | 不利于展示规范性 | 仅早期调试使用 |

默认建议：APScheduler + Cron。

### 8.7 部署选型

| 方案 | 优点 | 缺点 | 结论建议 |
|---|---|---|---|
| Docker Compose 单机 | 快速稳定、便于演示、可复制 | 扩展性一般 | 推荐（毕设首选） |
| 裸机进程部署 | 简单直接 | 环境复现差 | 不建议作为最终方案 |
| Kubernetes | 架构先进 | 学习与运维复杂度高 | 毕设不建议主线采用 |

默认建议：Docker Compose 单机。

## 9. 最终落地技术栈（已确认）

1. 前端：Vue 3 + Vite + ECharts。  
2. 后端：Spring Boot（Java 17，Spring Web，Spring Security，MyBatis-Plus）。  
3. 数据处理：PySpark（Spark SQL + Structured Streaming）。  
4. 数据库：MySQL 8.0。  
5. 消息队列：Kafka。  
6. 向量检索：FAISS（Python 服务化封装）。  
7. 调度：APScheduler + Cron。  
8. 部署：Docker Compose。

该方案兼顾技术完整性与毕设可实现性：前后端分离清晰、流式链路真实、识曲能力可演示、部署复现成本低。

## 10. 质量属性与可测指标

### 10.1 可扩展性

1. 新增数据源仅新增采集适配器，不影响核心服务。  
2. 新增指标通过 DWS/ADS 增表扩展。

### 10.2 性能

1. 榜单刷新延迟目标：<= 10 秒。  
2. API 平均响应：<= 2 秒。  
3. 识曲接口响应：<= 5 秒（样本规模内）。

### 10.3 可靠性

1. 采集任务失败自动重试。  
2. 任务运行日志可追踪。  
3. 核心功能可用率目标：>= 95%。

## 12. 风险登记与缓解

| 风险 | 等级 | 触发信号 | 缓解措施 |
|---|---|---|---|
| 数据源反爬或字段变化 | 高 | 采集失败率上升 | 多源备份、字段映射热更新 |
| 识曲准确率低 | 中 | Top1 命中率不达标 | 增强特征、调参与重排序 |
| 实时链路延迟偏高 | 中 | 榜单更新滞后 | 调整窗口参数、减少计算维度 |
| 开发时间不足 | 高 | 关键模块延期 | 严格执行 MoSCoW，优先 Must |

## 14. 模块技术映射与部署拓扑

### 14.1 模块技术映射

| 模块 | 技术实现 | 说明 |
|---|---|---|
| Web 前端 | Vue 3 + Vite + ECharts | 看板展示、筛选联动、识曲上传交互 |
| API 后端 | Spring Boot + Spring Security + MyBatis-Plus | 统一接口、鉴权、业务编排 |
| 实时处理 | Spark Structured Streaming + Kafka | 消费行为流并实时生成 TopN |
| 离线处理 | Spark SQL / MLlib | 分类、分群、画像与汇总计算 |
| 音频检索 | Python + Librosa + FAISS | 特征提取、向量索引、TopK 检索 |
| 存储 | MySQL + 文件存储（Parquet/CSV） | 结构化结果与离线明细持久化 |
| 调度 | APScheduler + Cron | 采集、清洗、重算任务定时执行 |

### 14.2 Docker Compose 部署拓扑（单机）

1. nginx：静态资源与反向代理入口。  
2. frontend：Vue 打包后的前端服务。  
3. api-service：Spring Boot 业务 API 服务。  
4. spark-job：Spark 批处理与流处理作业容器。  
5. kafka + zookeeper：行为流接入与缓冲。  
6. feature-service：Python FAISS 检索服务。  
7. mysql：维表、结果表、系统表。  
8. storage：挂载原始数据、清洗数据与音频样本目录。

### 14.3 关键交互链路

1. 采集模块写入 ODS，清洗后落地 DWD。  
2. 行为事件写入 Kafka，Spark Streaming 消费并更新榜单快照。  
3. Spark 批处理生成 DWS/ADS，API 直接查询展示。  
4. 识曲请求由 Spring Boot 转发至 feature-service，返回 TopK 候选。  
5. 前端通过 API 聚合查询榜单、画像、检索结果并渲染图表。
