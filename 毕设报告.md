# 基于Spark的在线音乐信息系统的设计与实现

## 摘要

在数字音乐内容迅速扩张的背景下，在线音乐平台每天都会产生大量结构化与半结构化数据，包括歌曲元数据、榜单热度、歌单聚合信息、歌手关联关系以及歌词文本等。传统以人工统计和静态报表为主的数据分析方式难以满足动态更新、跨主题联动和即时可视化展示的需求。本文围绕“网易云单平台闭环实现”这一研究目标，完成了一个基于 Spark 的在线音乐信息系统设计与实现。系统以数据采集、数据管理、数据服务和可视化展示为主线，构建了从公开数据抓取到业务页面呈现的完整工程链路。

在系统设计层面，本文采用“实体主表 + 关系映射表 + 分析结果表”的建模思想，避免按照业务页面拆分为冗余大宽表，显著降低重复存储和后续维护成本。核心实体包括 songs、artists、playlists、ranks、users，核心关系包括 rank_songs、playlist_songs、artist_songs、user_recent_songs，分析结果存放在 song_tags 与多类画像表中。针对标签生成依赖歌词文本的问题，系统在歌曲模型中引入 lyric_text 字段，并实现歌词采集、增量更新和持久化策略。为兼顾效率与数据完整性，歌词链路采用“默认仅补全空歌词，必要时全量刷新”的策略，减少重复请求并缩短每日任务耗时。

在实现层面，系统后端基于 Spring Boot 构建统一 REST 服务，前端基于原生 HTML 与 ECharts 实现三域联动界面，支持“推荐列表—歌曲列表—分析展示”交互流程。系统同时接入 DashScope OpenAI 兼容接口，实现歌词标签自动生成，并提供规则兜底机制，保障模型不可用场景下的功能连续性。部署层通过 Docker Compose 编排 MySQL、Kafka、采集作业、API 服务与前端容器，实现一键构建与启动。测试结果表明，该系统在毕业设计阶段已具备较完整的数据闭环能力与展示能力，能够稳定支撑排行榜、歌单和歌手三类业务的查询与分析。

关键词：Spark；在线音乐信息系统；数据采集；数据库设计；歌词标签；可视化分析

## Abstract

With the rapid expansion of digital music ecosystems, online music platforms continuously generate large volumes of structured and semi-structured data, including song metadata, ranking trends, playlist aggregations, artist-song relationships, and lyric content. Traditional static reporting is no longer sufficient for dynamic updates, cross-domain analysis, and interactive visualization. This thesis focuses on a single-platform closed-loop implementation and presents the design and implementation of an online music information system based on Spark.

At the architecture level, the system adopts a normalized model based on entity tables, relationship mapping tables, and analysis-result tables, instead of maintaining multiple redundant wide tables for different business pages. Core entities include songs, artists, playlists, ranks, and users, while major mappings include rank_songs, playlist_songs, artist_songs, and user_recent_songs. To support lyric-driven intelligence, lyric_text is integrated into the song schema, and a complete lyric ingestion and incremental update mechanism is implemented. The default strategy only fills missing lyrics and provides optional full refresh, balancing data quality and collection efficiency.

At the implementation level, the backend is developed with Spring Boot and provides unified REST APIs; the frontend is built with native web technologies and ECharts, supporting a three-domain workflow from recommendation lists to song details and analysis views. For automatic tag generation, the system integrates the DashScope OpenAI-compatible API and introduces fallback rules to ensure robustness when model invocation is unavailable. The full stack is containerized and orchestrated by Docker Compose, enabling one-command startup across data, service, and presentation layers. Experimental validation demonstrates that the system achieves a stable end-to-end process from crawling to storage, querying, and visual analytics, which satisfies the key requirements of the graduation project stage.

Key words: Spark; online music system; data crawling; database design; lyric tagging; visual analytics

## 第1章 引言

### 1.1 课题背景

在线音乐业务具有典型的数据密集和时效敏感特征。用户在短时间内对热门内容的关注变化会直接体现在榜单排名、播放行为和评论活跃度上，而歌曲本身又具有歌词文本、演唱者关系、所属歌单等天然关联属性。对这类场景进行信息化构建时，系统不仅要支持稳定采集，还要支持多维查询、快速联动和可解释展示。从工程实践角度看，若仅构建单一采集脚本而缺少统一的数据模型与服务层，会导致前后端耦合严重、数据口径不统一以及后续扩展困难。因此，本课题以“可运行、可复现、可演示”为核心目标，尝试构建一套适用于毕业设计场景的在线音乐分析系统。

### 1.2 研究目的与研究内容

本课题并非单纯完成某一个功能点，而是面向“数据闭环”进行系统化实现。研究重点包括四个方面：第一，建立面向网易云单平台的稳定采集机制，覆盖排行榜、歌曲、歌手及歌词信息；第二，构建标准化数据库模型，使实体、关系和分析结果之间具有明确边界；第三，实现统一 API 层，将数据能力转化为前端可消费的业务能力；第四，接入 AI 标签生成能力，使歌词内容能够转化为结构化标签并用于分析展示。围绕上述目标，本文完成了需求梳理、架构设计、模块实现、数据库落地、容器部署和测试验证等工作。

### 1.3 论文结构安排

本文按照毕业设计报告的逻辑顺序展开。第二章说明需求来源与需求分解；第三章说明系统整体架构和设计原则；第四章进入模块层面的详细设计；第五章重点描述数据库逻辑模型与物理模型；第六章给出关键实现过程；第七章展示测试方案与结果分析；第八章总结课题成果并提出后续改进方向。

## 第2章 需求分析

### 2.1 需求来源与边界约束

本课题需求来自《需求规格说明书》《需求调整说明》以及项目阶段性实现目标。相较于早期“多平台统一分析”设想，当前阶段明确聚焦网易云单平台，强调在有限时间内实现可验证的工程闭环。该边界定义并不意味着忽略扩展性，而是通过在模型和接口层预留字段与路径，为后续多平台接入提供结构基础。系统数据来源限定为公开可访问信息，设计上遵循最小化采集原则，不存储敏感隐私数据，保证课题实施的合规性与可控性。

### 2.2 业务需求分析

从业务视角看，系统目标不是“展示某一张榜单”，而是支持三个业务域的统一分析能力。排行榜域强调热点变化和排序结构，歌单域强调集合语义与内容聚合，歌手域强调作品结构和代表曲信息。三个业务域最终要落在同一条交互路径上，即用户先看到推荐对象，再查看对象关联的歌曲列表，最后进入分析区域查看歌词、标签与相似推荐。该交互路径要求后端数据具备可追溯关系，要求前端接口具备一致语义，也要求数据库模型能够承载多对多映射而不是硬编码字段。

### 2.3 功能需求分析

功能需求可归纳为数据能力、服务能力和展示能力三个层面。数据能力要求系统能稳定完成榜单、歌曲、歌手与歌词采集，并形成可持续更新机制；服务能力要求提供推荐、明细、分析和标签生成接口；展示能力要求将接口结果转化为可视化和联动交互。特别是在标签能力方面，需求明确提出“基于歌词调用 AI 生成 tags 并持久化”，这意味着系统不仅要完成 AI 调用本身，还要设计标签存储模型、版本字段和回退机制。由于外部模型调用存在不可用风险，系统在设计阶段同步引入规则兜底，保证关键链路不会因模型异常而中断。

### 2.4 非功能需求分析

系统在非功能层面主要关注可部署性、可维护性和可扩展性。可部署性要求项目能够在普通开发机上一键启动，因此采用 Docker Compose 对多服务进行统一编排。可维护性要求代码按职责分层，数据库字段命名与索引策略一致，任务日志可追踪。可扩展性要求在不推翻现有模型前提下支持新增数据域与分析能力，因此数据表设计遵循主实体与映射关系分离原则。性能方面，本课题不追求生产级高并发指标，但需保证常用接口在演示环境可稳定响应。

## 第3章 系统总体设计

### 3.1 总体架构设计

系统整体采用“采集层—处理层—存储层—服务层—展示层”的分层架构。采集层负责访问公开接口并将原始结果标准化，处理层负责补充实时榜单与行为流数据，存储层负责统一管理实体关系与分析结果，服务层将数据能力封装为稳定 API，展示层负责交互和图表呈现。该架构的核心价值在于每层职责明确，避免“前端直接拼 SQL”或“采集脚本直接面向页面结构”这类高耦合实现方式。通过分层设计，系统在功能迭代时可局部修改，例如新增标签维度时主要影响分析层和服务层，而不会破坏采集主链路。

### 3.2 技术路线设计

技术路线选择遵循“可实现、可验证、可迁移”的原则。数据采集与调度使用 Python，便于快速迭代与接口适配；实时行为链路保留 Spark + Kafka，用于体现流式处理能力并提供实时榜单演示数据；结构化存储选用 MySQL 8.0，兼顾事务一致性与 SQL 可读性；服务端采用 Spring Boot 3，统一管理接口、配置和数据库访问；前端使用原生 HTML + ECharts，降低工程复杂度并突出数据展示逻辑；部署层使用 Docker Compose，确保评审环境的复现效率。整体技术组合在毕业设计场景下兼顾了工程性和学习成本。

### 3.3 数据流与控制流设计

系统数据流从采集任务开始，采集程序先请求榜单列表，再逐个榜单请求歌曲明细，同时抽取歌手关系并写入数据库。对于歌曲信息，系统会检查歌词字段是否为空，若为空则调用歌词接口补全文本。数据入库后，服务层对外提供推荐和分析接口，前端通过域切换加载对应推荐项，并根据用户点击请求歌曲明细和分析结果。在 AI 标签场景下，前端触发标签生成请求，服务层调用模型接口并回写 song_tags，再由分析接口读取最新标签结果反馈页面。控制流方面，任务执行状态统一写入 sys_task_log，用于故障定位与运行审计。

### 3.4 部署架构设计

部署架构采用容器化方案，核心服务包括 MySQL、Zookeeper、Kafka、Spark 作业容器、网易云采集容器、API 服务和前端容器。容器间通过 Compose 网络通信，数据库作为共享状态中心。采集容器按日循环执行采集命令，API 容器负责对外暴露业务接口，前端容器提供静态页面访问入口。该部署方式使系统在本地与演示环境保持一致运行方式，避免“本地可跑、迁移失败”的环境差异问题。

## 第4章 模块详细设计

### 4.1 数据采集模块详细设计

数据采集模块由 `collect_netease_platform_data.py` 主脚本实现，其核心流程包括榜单抓取、歌曲标准化、歌手映射和歌词补全。脚本首先访问榜单接口获取榜单集合，随后以榜单 ID 为参数请求歌曲明细，并将返回字段映射为统一内部结构。歌曲标准化阶段对 song_id、song_name、album_name、publish_time 等关键字段进行裁剪和类型处理，确保写库时字段口径一致。歌手映射阶段将单曲中的多个歌手抽取为独立关系记录，写入 artist_songs 表。歌词补全阶段在默认模式下先查询数据库已有歌词集合，仅对缺失歌词的 song_id 调用歌词接口，从而减少冗余请求。脚本最终将执行状态、异常信息和耗时写入 sys_task_log，形成任务级可追踪记录。

### 4.2 实时处理模块详细设计

实时处理模块由 `streaming_rank_job.py` 承担，主要用于演示流式链路。该模块通过 Kafka 生产模拟播放事件，并由消费端实时聚合生成 minute 级榜单快照。该链路与主采集链路并行存在，不直接替代正式榜单数据，而是提供“实时热度变化”展示能力。数据写入 fact_play_event 与 fact_rank_snapshot 后，前端可通过榜单接口读取 minute 数据绘制热度图。该模块在教学演示中体现了 Spark 与消息队列结合的处理思想，并为后续接入真实行为数据预留了接口模式。

### 4.3 后端服务模块详细设计

后端服务层围绕业务语义而非单表查询进行接口组织。推荐接口分别对应排行榜、歌单和歌手三个域，负责输出具备业务可读性的对象列表；详情接口负责输出对象关联歌曲，按位置、时间或热度排序；分析接口负责输出单曲歌词、标签分布和相似歌曲。为了保证接口稳定性，服务层在查询中统一进行参数边界处理，并将聚合逻辑放在服务内部，避免前端承担数据拼装责任。在标签生成链路中，服务层先读取歌曲歌词与标题，调用 AI 模型生成标签文本，再将结果解析为标准数组并落库。若模型返回异常或格式不可解析，系统自动执行规则兜底并仍然产出标签结果，从而保证业务连续性。

### 4.4 AI 标签模块详细设计

AI 标签模块的输入是歌曲名与歌词文本，输出是最多若干个结构化标签及其得分。模块在实现中考虑了三个工程问题。第一，模型配置不写死在代码中，而是通过环境变量注入，便于后续替换模型或迁移平台。第二，模型输出可能存在格式不稳定问题，因此在解析层加入“提取 JSON 数组 + 文本降级分割”双重策略，提高成功率。第三，标签更新不能破坏已有有效数据，因此在写库时保留来源模型字段，并通过 upsert 机制覆盖同模型历史结果。该模块不仅满足标签生成需求，也为后续扩展到情感标签、风格标签或多模型对比提供了结构基础。

### 4.5 前端展示模块详细设计

前端模块采用三栏联动布局。顶栏用于业务域切换，左侧展示推荐对象，中间展示歌曲列表与趋势图，右侧展示歌词、标签和相似歌曲。该布局对应需求文档中的“推荐列表—详情列表—分析展示”流程，用户路径清晰，评审演示效率高。页面通过统一 fetch 封装调用后端 API，在每次域切换后自动刷新推荐与歌曲明细，并默认选中首条歌曲进行分析。标签生成按钮触发后，页面会重新拉取分析接口，确保用户立即看到最新标签结果。任务日志区域持续刷新，可作为系统运行健康度的可视化观测窗口。

### 4.6 调度与部署模块详细设计

调度模块通过 `netease-collector` 容器执行循环命令实现每日自动采集。容器启动后先执行一次采集任务，随后休眠固定时长再继续下一轮，从而在不引入额外调度中间件的前提下完成日级任务调度。部署模块由 docker-compose.yml 统一维护，包括环境变量、端口映射、容器依赖关系和健康检查。API 服务通过环境变量读取数据库连接和 AI 配置，实现配置与代码分离。该方案在毕业设计阶段足以支撑稳定演示，并具备迁移到正式调度平台的可升级空间。

## 第5章 数据库设计

### 5.1 数据库设计目标

数据库设计以“去冗余、可追溯、易扩展”为原则。去冗余要求相同歌曲只存储一份主数据，避免在排行榜、歌单、歌手三个场景中重复写入。可追溯要求所有关系均有明确映射表和时间字段，确保能够回溯某次采集写入。易扩展要求模型在单平台阶段就预留版本、来源和可扩展字段，以便后续多平台接入时尽量减少结构性改造。

### 5.2 逻辑模型设计

系统逻辑模型由实体、关系和分析结果三部分组成。实体层描述稳定对象属性，如歌曲名称、歌手名称、歌单名称等；关系层描述对象之间的多对多关联，如某榜单在某时间对应哪些歌曲、某歌手对应哪些歌曲；分析结果层描述算法输出，如歌曲标签、画像结果和版本信息。该逻辑模型避免了“按页面建表”的临时做法，使业务语义在数据层得到稳定表达。

### 5.3 物理表设计说明

songs 表以 song_id 作为主键，除基础字段外新增 lyric_text 用于承载歌词全文。artists、playlists、ranks、users 分别以平台对象 ID 为主键。rank_songs、playlist_songs、artist_songs 等映射表通过组合唯一键控制重复写入，并保留 crawl_time 等时间字段支撑历史分析。song_tags 表存储标签名称、得分、来源模型和版本字段，既能支持当前展示，也能支持模型对比。sys_task_log 记录任务名、状态、耗时和摘要消息，是系统运维与排障的重要依据。

### 5.4 索引与约束设计

考虑常见查询路径，系统对歌曲名、榜单时间、榜单排名、更新时间等字段建立索引。映射表中的组合唯一键保证了幂等写入与关系唯一性，防止采集重复造成脏数据累积。分析表中保留 source_model 与 source_version 字段，使标签与画像可追溯到具体生成配置。整体约束策略强调“写入可控、查询高效、历史可追踪”三项平衡。

### 5.5 数据更新与一致性策略

采集写入统一采用 upsert 语义。对于歌词字段，系统在更新时采用 `COALESCE(VALUES(lyric_text), lyric_text)` 策略，避免空值覆盖已有内容。对于标签数据，系统先按模型来源清理旧结果再写入新结果，保证同一模型下结果一致。对于关系数据，依赖唯一键防止重复。通过以上策略，系统在高频增量采集下仍能保持主数据稳定与分析数据可解释。

## 第6章 系统实现与关键技术

### 6.1 关键实现一：歌词链路端到端落地

在本课题中，歌词不仅是展示文本，更是 AI 标签生成的核心输入，因此其链路完整性直接决定分析模块可用性。实现过程中，首先在数据库层补充 lyric_text 字段，并同步更新初始化脚本、历史兼容表和采集写入 SQL。其次在采集层引入歌词接口请求函数，结合“已有歌词集合查询”减少重复请求。最后在 API 层将歌词作为歌曲分析响应的一部分返回前端。该链路实现后，系统从“只展示榜单”升级为“支持内容语义分析”的形态。

### 6.2 关键实现二：AI 标签生成与回退机制

标签生成接口由后端统一封装。调用模型前，系统会截取适度长度的歌词文本，降低请求体体积并控制调用成本。模型返回后，系统先尝试解析 JSON 数组；若解析失败，则转为文本分隔并进行标签清洗，最终限制标签数量和长度。若模型完全不可用，系统根据关键词规则输出基础标签，避免界面出现空分析状态。该机制体现了“功能优雅降级”思想，是毕业设计中将 AI 能力与工程稳定性结合的关键实践。

### 6.3 关键实现三：三域联动前端交互

前端页面并未将三类业务拆成多个孤立页面，而是通过统一状态管理在同一页面中切换域类型、推荐对象和分析歌曲。这种实现方式显著提升了演示连贯性。用户每次切换业务域后，系统自动请求对应推荐列表并加载首个对象明细；点击任一歌曲可立即查看歌词、标签分布和相似歌曲。由于后端接口语义统一，前端无需针对每个域编写完全不同的数据适配逻辑，降低了维护复杂度。

### 6.4 关键实现四：容器化一键部署

项目部署通过 Docker Compose 统一管理，避免了本地环境差异造成的运行失败。实际验证表明，执行 `docker compose up -d --build` 后可自动完成镜像构建和服务启动，前后端页面与接口可直接访问。采集容器与 API 容器通过内部网络访问同一数据库，日志可通过 compose 命令集中查看。该部署方式使系统在答辩演示时具备较高的可复现性。

## 第7章 系统测试与结果分析

### 7.1 测试方案

测试方案分为静态校验、功能校验、联调校验和部署校验四个层次。静态校验用于确认脚本语法和后端构建是否通过；功能校验用于验证采集、查询、分析和标签生成链路；联调校验用于确认前后端交互逻辑与数据展示一致；部署校验用于确认容器编排后的服务状态和访问可用性。该分层方案有助于在出现问题时快速定位故障层级。

### 7.2 功能测试结果

在数据采集方面，网易云榜单、歌曲、歌手与歌词可稳定入库，任务日志可记录执行结果与异常摘要。在接口层面，推荐与详情接口可按参数返回数据，歌曲分析接口可返回歌词、标签和相似歌曲结果。在 AI 能力方面，标签生成接口可调用外部模型并写入 song_tags，当模型不可用时可触发规则兜底并保持接口可用。前端联调结果显示，三域切换、歌曲选择、标签刷新和图表渲染均符合预期。

### 7.3 性能与稳定性观察

在演示环境下，常用查询接口保持秒级响应，页面刷新体验稳定。采集任务由于采用“仅补空歌词”策略，日常运行耗时较全量更新模式明显下降。系统在连续运行期间未出现影响主链路的致命错误，任务异常可通过 sys_task_log 快速定位。整体上，系统满足毕业设计阶段“稳定可演示”的性能与可用性要求。

### 7.4 局限性分析

尽管系统已形成完整闭环，但仍存在若干局限。第一，当前歌单与听众画像仍以结构预留为主，深度分析能力有待进一步实现。第二，标签质量受模型输出与歌词质量影响，尚缺乏系统化评估指标。第三，单机容器化部署满足教学演示，但不具备生产级高并发与高可用能力。上述局限并不影响课题目标达成，但为后续迭代指明了方向。

## 第8章 结论与展望

本课题围绕“基于 Spark 的在线音乐信息系统”完成了从需求分析到工程落地的全过程实现。通过单平台聚焦策略，系统在有限周期内实现了数据采集、数据库建模、服务接口、前端展示、AI 标签生成和容器部署的完整闭环。实践结果表明，采用实体与关系分离的数据建模方法能够有效支撑多业务域联动展示；引入歌词字段和标签生成能力能够显著提升系统分析深度；通过容器化部署可显著提高系统复现效率和演示稳定性。总体而言，课题达到毕业设计预期目标，具备较好的工程完整性和可验证性。

后续工作将围绕三个方向展开。其一是提升分析深度，完善歌单画像、歌手画像和听众画像的自动化生成链路。其二是提升智能质量，建立标签评估指标并引入多模型对比策略。其三是提升系统工程能力，逐步引入更标准的调度体系与更细粒度的服务治理机制，以支持多平台与更复杂业务场景。
