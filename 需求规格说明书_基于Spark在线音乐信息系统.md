## 2. 项目概述

### 2.1 项目背景

在线音乐平台产生了大量歌曲元数据、用户行为数据和音频内容数据。传统静态分析无法满足热点变化快、用户偏好多样化、歌曲检索场景实时化的需求。本项目通过 Spark 构建离线与实时结合的数据处理能力，形成可展示、可验证的完整系统原型。

### 2.2 项目目标

1. 构建音乐数据采集、清洗、治理、存储的完整流程。  
2. 实现歌曲、歌手、播放行为的实时统计与榜单排行。  
3. 完成音乐分类、听众分类与群体画像分析。  
4. 实现听歌识曲与哼唱搜歌的相似歌曲检索能力。  
5. 提供可视化展示页面用于分析和答辩演示。

### 2.3 项目范围

#### 2.3.1 范围内（In Scope）

1. 多平台公开数据采集（QQ音乐、酷狗、抖音等）。  
2. 数据清洗、标准化、实体对齐。  
3. 实时排行与多维统计分析。  
4. 音乐/听众分类与群体画像。  
5. 音频特征检索（听歌识曲、哼唱搜歌）。  
6. 数据可视化与系统管理后台。

#### 2.3.2 范围外（Out of Scope）

1. 商业级版权分发与在线完整播放服务。  
2. 超大规模生产集群的高可用部署。  
3. 复杂商业推荐闭环（广告、付费转化联动）。

## 3. 总体描述

### 3.1 产品定位

本系统定位为“基于 Spark 的在线音乐分析与智能检索教学演示平台”，强调技术链路完整和核心场景可用。

### 3.2 系统边界

1. 输入：平台公开元数据、行为日志、音频样本。  
2. 处理：清洗治理、实时计算、离线建模、检索匹配。  
3. 输出：榜单、画像、检索结果、分析图表、任务日志。

### 3.3 运行环境约束

1. 可运行于单机或小型集群。  
2. 支持 Linux/macOS 开发环境。  
3. 需具备 Spark、Python、数据库、Web 前端运行条件。  
4. 使用公开可访问数据，遵守平台协议和法律法规。

## 4. 业务需求

### 4.1 业务目标

1. 通过实时排行发现音乐热度变化。  
2. 通过分类与画像支持人群洞察。  
3. 通过识曲与哼唱检索提升歌曲发现能力。  
4. 通过可视化支撑教学展示和结果验证。

### 4.2 业务流程（高层）

1. 数据采集：定时抓取平台公开信息。  
2. 数据治理：清洗去重、标准化与对齐。  
3. 数据计算：离线统计 + 实时流式聚合。  
4. 分析建模：分类与画像、音频特征索引。  
5. 结果服务：查询接口与可视化展示。  
6. 运维监控：任务调度、日志、告警与重试。

### 4.3 业务规则

1. 同一歌曲跨平台需进行实体映射，保证统一 song_id。  
2. 榜单按时间窗口计算，窗口参数可配置。  
3. 听歌识曲/哼唱搜歌返回候选列表与置信度。  
4. 统计指标口径统一，所有图表遵循同一指标定义。  
5. 采集仅限公开信息，不采集敏感个人隐私。

## 5. 功能需求

### 5.1 功能模块清单

| 模块ID | 模块名称 | 说明 |
|---|---|---|
| FR-01 | 数据采集模块 | 多平台歌曲/歌手/热度数据采集 |
| FR-02 | 数据清洗模块 | 去重、补全、格式统一、异常处理 |
| FR-03 | 数据存储模块 | 分层存储与查询服务 |
| FR-04 | 实时排行模块 | 实时 TopN 榜单与趋势分析 |
| FR-05 | 音乐分类模块 | 风格/标签分类 |
| FR-06 | 听众分类模块 | 用户群体划分与特征提取 |
| FR-07 | 群体画像模块 | 群体偏好建模和对比 |
| FR-08 | 相似歌曲检索模块 | 基于特征向量的相似检索 |
| FR-09 | 听歌识曲模块 | 音频片段识别 |
| FR-10 | 哼唱搜歌模块 | 哼唱旋律检索 |
| FR-11 | 可视化模块 | 图表看板和分析页面 |
| FR-12 | 系统管理模块 | 登录鉴权、任务日志、参数配置 |

### 5.2 详细功能需求

#### FR-01 数据采集模块

1. 系统应支持配置化数据源接入（平台、字段、采集频率）。  
2. 系统应支持定时采集与手动触发采集。  
3. 系统应记录每次采集任务的开始时间、结束时间、结果状态。  
4. 系统应支持失败重试机制（至少 3 次重试可配置）。

验收标准：可成功采集不少于 3 个平台的数据；任务日志完整可查。

#### FR-02 数据清洗模块

1. 系统应支持主键去重和重复记录合并。  
2. 系统应支持字段标准化（时间、数值、标签格式）。  
3. 系统应支持缺失值策略处理（填充、丢弃、标记）。  
4. 系统应支持异常数据识别并输出异常报告。

验收标准：清洗后数据重复率下降至可接受范围，输出质量统计报告。

#### FR-03 数据存储模块

1. 系统应采用分层存储策略（原始层、清洗层、服务层）。  
2. 系统应支持历史数据归档和增量写入。  
3. 系统应提供统一数据访问接口供分析和可视化调用。

验收标准：任一图表可追溯至数据来源层和处理批次。

#### FR-04 实时排行模块

1. 系统应基于流式数据进行分钟级榜单更新。  
2. 系统应支持日榜、周榜、月榜切换。  
3. 系统应支持按歌曲、歌手、分类维度查看排行。  
4. 系统应支持热度增速指标计算。

验收标准：连续输入数据时榜单可稳定刷新；支持多时间窗口切换。

#### FR-05 音乐分类模块

1. 系统应支持规则或模型方式进行歌曲风格分类。  
2. 系统应支持多标签分类并展示标签权重。  
3. 系统应支持分类结果回溯到原始标签依据。

验收标准：分类结果可视化展示，且可解释分类依据。

#### FR-06 听众分类模块

1. 系统应根据行为特征进行用户分群。  
2. 系统应支持活跃度、偏好类型、时段行为等维度。  
3. 系统应支持分群结果导出与复用。

验收标准：形成至少 3 类听众群体并可展示特征差异。

#### FR-07 群体画像模块

1. 系统应生成群体画像标签（偏好风格、活跃时段、热门歌手）。  
2. 系统应支持群体之间的对比分析。  
3. 系统应支持按时间范围查看画像变化趋势。

验收标准：可展示画像雷达图或对比图并支持筛选。

#### FR-08 相似歌曲检索模块

1. 系统应支持音频特征提取与向量化存储。  
2. 系统应支持输入歌曲返回 TopK 相似歌曲。  
3. 系统应显示相似度分值和候选排序。

验收标准：给定歌曲可返回合理候选列表，结果可重复验证。

#### FR-09 听歌识曲模块

1. 系统应支持上传音频片段进行识别。  
2. 系统应返回候选歌曲列表和置信度。  
3. 系统应支持识别失败提示和重试机制。

验收标准：在测试集上达到预设命中率阈值。

#### FR-10 哼唱搜歌模块

1. 系统应支持上传哼唱音频并提取旋律特征。  
2. 系统应返回旋律相似候选歌曲。  
3. 系统应支持显示检索耗时和结果置信区间。

验收标准：在演示样本上可稳定返回可解释候选结果。

#### FR-11 可视化模块

1. 系统应提供概览看板（总量、趋势、实时榜单）。  
2. 系统应提供分类分布、画像分析、检索结果展示。  
3. 系统应支持按时间、平台、分类维度联动筛选。  
4. 系统应支持图表导出。

验收标准：至少包含 6 类可交互图表并支持联动筛选。

#### FR-12 系统管理模块

1. 系统应支持账号登录与角色权限控制。  
2. 系统应支持任务管理（启动、停止、重试、查看日志）。  
3. 系统应支持参数配置（榜单权重、窗口大小、阈值）。  
4. 系统应支持审计日志记录。

验收标准：管理员可独立完成任务维护与参数调整。

## 6. 非功能需求

### 6.1 性能需求

1. 实时榜单更新延迟应不高于 10 秒（演示环境）。  
2. 查询类接口平均响应时间应不高于 2 秒。  
3. 系统应支持至少 50 并发查询请求的稳定响应（演示级）。

### 6.2 可用性需求

1. 核心功能可用率应达到 95% 以上（验收周期）。  
2. 出现任务失败时应支持自动重试与手动补偿。

### 6.3 安全与合规需求

1. 系统应仅采集公开可访问数据。  
2. 系统不得存储敏感个人隐私信息。  
3. 系统应提供基础访问控制和操作审计。

### 6.4 可维护性需求

1. 各模块应解耦并具备独立配置能力。  
2. 日志应统一格式并支持关键链路追踪。  
3. 配置变更应可追踪、可回滚。

### 6.5 可扩展性需求

1. 新增数据源时无需大规模改造主流程。  
2. 新增分类标签和画像维度时可配置化扩展。  
3. 检索算法应支持替换或并行对比。

## 7. 数据需求

### 7.1 数据来源

1. QQ音乐公开信息。  
2. 酷狗音乐公开信息。  
3. 抖音音乐相关公开信息。  
4. 公开可用音频样本或自建演示样本集。

### 7.2 数据实体与关键字段

| 实体 | 关键字段 |
|---|---|
| 歌曲 | song_id, song_name, artist_id, duration, tags, publish_time |
| 歌手 | artist_id, artist_name, region, style |
| 行为日志 | user_id, song_id, play_time, play_count, like_count, comment_count |
| 音频特征 | song_id, mfcc_vector, chroma_vector, tempo, embedding |
| 榜单快照 | rank_time, rank_type, item_id, score, rank_position |

### 7.3 数据质量要求

1. 主键唯一率应达到 99% 以上。  
2. 关键字段缺失率应低于 5%。  
3. 异常记录应单独标记并可回溯。  
4. 数据口径需一致并输出口径定义文档。

## 8. 外部接口需求

### 8.1 数据采集接口

1. 支持 HTTP/HTTPS 请求抓取公开数据。  
2. 支持请求频率控制与异常重试。  
3. 支持采集字段映射配置。

### 8.2 系统服务接口

1. 提供榜单查询接口。  
2. 提供分类和画像查询接口。  
3. 提供识曲/哼唱检索接口。  
4. 接口返回统一 JSON 结构。

### 8.3 可视化交互接口

1. 前端应支持按条件组合查询。  
2. 前端应支持图表联动请求。  
3. 前端应支持结果分页和导出。

## 9. 约束条件

1. 项目周期受毕业设计时间限制，优先完成核心功能。  
2. 数据量以“可支撑演示和实验分析”为目标，不追求生产规模。  
3. 计算与存储资源受实验环境限制，采用轻量化部署方案。  
4. 严格遵循数据合规和学术诚信要求。

## 12. 风险与应对

| 风险 | 影响 | 应对策略 |
|---|---|---|
| 反爬限制 | 数据不稳定 | 多源冗余、限速、缓存与重试 |
| 数据质量波动 | 分析失真 | 质量规则、异常告警、人工抽检 |
| 识曲效果不佳 | 演示体验下降 | 基线模型先行、特征优化与重排序 |
| 资源不足 | 性能不达标 | 控制样本规模、优化任务并行度 |
| 时间紧张 | 功能不完整 | 严格按 MoSCoW 控制范围 |

## 14. 附录

### 14.1 指标口径建议

1. 热度得分 = 播放量权重 + 点赞权重 + 评论权重 + 增速权重。  
2. 活跃用户定义：在统计周期内播放次数达到阈值。  
3. 新歌定义：发布时间在近 N 天内的歌曲。

### 14.2 建议验收材料清单

1. 系统演示视频。  
2. 样本数据说明。  
3. 测试报告（功能、性能、准确性）。  
4. 部署说明与运行截图。
